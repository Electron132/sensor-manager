# Sensor Manager 项目计划

## 1. 项目定位

这是一个使用 C++ 编写的命令行传感器管理器，用来练习从需求分析、类设计、容器使用、用户输入、错误处理，到代码拆分和文件保存的完整开发过程。

第一版不追求真实连接 STM32 或 ESP32，而是先用程序模拟传感器数据。项目重点是把 C++ 基础真正组织成一个可以运行、可以修改、可以解释的小型工程。

## 2. 最终目标

程序能够管理多个传感器，并支持以下操作：

- 添加传感器
- 查看全部传感器
- 根据 ID 查询传感器
- 修改传感器数据
- 修改传感器在线状态
- 删除传感器
- 保存传感器数据
- 从文件加载传感器数据
- 对非法输入给出提示，而不是直接崩溃

最终程序应当能够作为后续嵌入式项目中的“设备管理模块”原型。

## 3. 第一版范围

第一版只使用内存保存数据，程序退出后数据可以丢失。

第一版使用：

- C++17
- `class`
- `enum class`
- `std::vector`
- `std::string`
- 函数和引用
- `const`
- 枚举、循环和条件判断
- CMake
- CLion 调试器

第一版暂不加入：

- 图形界面
- 数据库
- 网络通信
- 多线程
- 智能硬件真实通信
- AI 功能
- 复杂的设计模式

## 4. 数据模型

每个传感器至少包含以下信息：

| 字段 | 类型建议 | 说明 |
|---|---|---|
| ID | `int` | 唯一标识，不能重复 |
| 名称 | `std::string` | 例如 Indoor Temperature |
| 类型 | `enum class` | 温度、湿度、光照等 |
| 数值 | `double` | 当前测量值 |
| 单位 | `std::string` | 例如 C、%、lux |
| 在线状态 | `bool` | 在线或离线 |
| 描述 | `std::string`，可选 | 对传感器的补充说明 |

建议先设计一个 `Sensor` 类，负责表示单个传感器；传感器集合和菜单逻辑可以先放在 `main.cpp` 中，后续再拆分。

## 5. 开发阶段

### 阶段 0：项目环境确认

目标：确认项目可以正常编译和运行。

任务：

- 确认 CLion 的 Toolchain、编译器和 CMake 配置正常
- 确认 `C++17` 生效
- 让程序能够启动并正常退出
- 在 CLion 中设置一个断点并成功调试
- 建立 Git 初始提交，可选但建议完成

完成标准：

- 项目能够在 CLion 中 Build
- 程序能够 Run
- Debug 模式能够进入断点
- 不因为环境问题影响后续开发

### 阶段 1：创建 `Sensor` 类

目标：把一个传感器抽象成一个 C++ 对象。

任务：

- 定义传感器的成员数据
- 设计构造函数
- 创建一个传感器对象
- 输出这个对象的信息
- 思考哪些成员应该是私有的
- 思考哪些操作应该通过成员函数完成

完成标准：

- 能创建至少一个传感器对象
- 能正确初始化所有必要字段
- 能输出传感器的完整信息
- 能解释类、对象、成员变量、成员函数和构造函数的作用

### 阶段 2：保存和展示多个传感器

目标：使用容器管理多个对象。

任务：

- 创建 `vector<Sensor>`
- 添加至少三个不同类型的传感器
- 遍历并打印全部传感器
- 处理传感器列表为空的情况
- 使用合适的 `const` 和引用，避免不必要的对象复制

完成标准：

- 能保存三个以上传感器
- 能一次性展示全部传感器
- 空列表时程序有明确提示
- 能解释为什么某些函数参数使用引用或 `const` 引用

### 阶段 3：实现添加功能

目标：让用户通过命令行输入传感器信息。

任务：

- 读取用户输入的 ID、名称、类型、数值和单位
- 检查 ID 是否已经存在
- 处理不同传感器类型
- 新传感器默认设置为在线或让用户选择状态
- 添加成功后给出明确提示

完成标准：

- 用户可以通过菜单添加传感器
- 重复 ID 不会覆盖已有传感器
- 添加成功后可以在列表中看到新传感器
- 输入不完整或非法时，程序不会直接崩溃

### 阶段 4：实现查询功能

目标：根据 ID 找到指定传感器。

任务：

- 输入传感器 ID
- 遍历传感器集合
- 找到后输出完整信息
- 找不到时给出提示
- 设计一个清晰的查找逻辑，避免在多个功能中重复写相同代码

完成标准：

- 能查询存在的传感器
- 能处理不存在的 ID
- 查找逻辑可以被更新和删除功能复用
- 能解释返回引用、指针或迭代器时各自的含义

### 阶段 5：实现更新功能

目标：修改已有传感器的数据。

任务：

- 根据 ID 找到目标传感器
- 更新测量数值
- 更新在线状态
- 可选：更新传感器名称和单位
- 查询不存在时不修改任何数据

完成标准：

- 修改后重新查看列表，数据已经更新
- 修改不存在的传感器时，程序状态不受影响
- 能处理数值输入错误
- 能解释为什么更新函数需要允许修改容器中的对象

### 阶段 6：实现删除功能

目标：从容器中删除指定传感器。

任务：

- 根据 ID 查找传感器
- 找到后删除
- 删除前可以要求用户确认
- 删除后验证列表内容
- 删除不存在的传感器时给出提示

完成标准：

- 能删除指定传感器
- 删除后该传感器不会再次出现在列表中
- 不会误删其他传感器
- 能处理空列表和不存在的 ID

### 阶段 7：完善菜单和输入处理

目标：让程序能够稳定地连续运行。

建议菜单：

```text
1. Add sensor
2. List sensors
3. Find sensor
4. Update sensor
5. Delete sensor
0. Exit
```

任务：

- 使用循环持续显示菜单
- 用户选择 `0` 时正常退出
- 非法菜单选项不导致程序崩溃
- 每项操作完成后返回菜单
- 为每项操作提供清晰的成功或失败提示

完成标准：

- 能连续执行多项操作
- 能从错误输入中恢复
- 退出时没有异常终止
- 用户不需要重新启动程序才能进行下一次操作

### 阶段 8：代码拆分

目标：从“能运行的单文件程序”过渡到更接近工程的结构。

建议结构：

```text
sensor-manager/
├── CMakeLists.txt
├── PLAN.md
├── README.md
├── include/
│   └── Sensor.h
└── src/
    ├── Sensor.cpp
    └── main.cpp
```

拆分原则：

- `Sensor.h` 放类的声明
- `Sensor.cpp` 放 `Sensor` 成员函数的实现
- `main.cpp` 负责菜单、输入和程序流程
- `CMakeLists.txt` 同时列出所有 `.cpp` 文件

完成标准：

- 拆分后程序功能与之前一致
- 项目仍然可以正常 Build 和 Run
- 头文件能被正确包含
- 能解释声明和实现为什么要分开

### 阶段 9：文件保存与加载

目标：让程序退出后仍能保留传感器数据。

任务：

- 设计简单的数据保存格式
- 程序启动时尝试加载文件
- 添加、修改、删除后保存数据
- 处理文件不存在的情况
- 处理文件内容格式错误的情况
- 退出前保存一次数据

可以先使用简单的文本格式，例如每行保存一个传感器；暂时不需要数据库或 JSON 库。

完成标准：

- 程序退出后重新启动，之前的数据仍然存在
- 文件不存在时程序能够正常启动
- 文件读取失败时能够给出提示
- 保存和加载后数据没有明显丢失

### 阶段 10：测试、重构和文档

目标：把项目整理成可以展示和复盘的作品。

测试场景至少包括：

- 空列表查看
- 正常添加传感器
- 添加重复 ID
- 查询存在和不存在的 ID
- 更新存在和不存在的传感器
- 删除存在和不存在的传感器
- 非法菜单输入
- 非法数值输入
- 文件不存在
- 文件内容异常

整理内容：

- 删除重复代码
- 统一命名风格
- 给关键函数补充简短注释
- 更新 `README.md`
- 记录遇到过的 bug 和解决方法
- 记录自己没有使用 AI 也能解释的代码部分

完成标准：

- 主要功能都能稳定运行
- 项目结构清晰
- README 能说明项目目标、编译方法和使用方法
- 你能够独立讲清楚项目的类设计和数据流动

## 7. 开发纪律

- 每次只实现一个功能
- 每完成一个小功能就运行和测试
- 不要一次生成整个项目代码
- 遇到问题先阅读错误信息，再定位到具体代码
- 可以向 AI 询问概念和错误原因，但先自己尝试实现
- 每个阶段留下一个 Git 提交或一条开发记录
- 不为了增加功能而增加功能，先保证已有功能稳定
